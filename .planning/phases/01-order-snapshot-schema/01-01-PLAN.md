**Key constraints from RESEARCH.md:**
- Snapshot va nella STESSA transazione di submit-order (rollback atomico)
- Usare DOUBLE PRECISION per amounts (non stringhe come in order_records legacy)
- Skip snapshot per warehouse orders (orderId.startsWith('warehouse-'))
- Lo snapshot è immutabile — non viene MAI aggiornato dopo la creazione
- Usare `pool.withTransaction()` pattern esistente
- Batch insert per snapshot items (pattern identico a order_articles)

**Pitfalls da evitare (da RESEARCH.md):**
- NON salvare amounts come stringhe — usare DOUBLE PRECISION
- NON aggiungere colonne a order_articles — lo snapshot deve essere separato e immutabile
- NON usare JSON per gli articoli — righe separate per query SQL dirette in Phase 2
- Skip warehouse orders e NC (per Phase 2, ma lo schema deve supportarlo)
</context>

Creare la migration SQL con due tabelle:

**Tabella 1: `agents.order_verification_snapshots`** (header dello snapshot)
- `id` SERIAL PRIMARY KEY
- `order_id` TEXT NOT NULL — FK logica verso order_records.id
- `user_id` TEXT NOT NULL — FK logica verso order_records.user_id
- `global_discount_percent` DOUBLE PRECISION — sconto globale ordine (nullable, non tutti gli ordini hanno sconto)
- `expected_gross_amount` DOUBLE PRECISION NOT NULL — totale lordo atteso (prima dello sconto globale)
- `expected_total_amount` DOUBLE PRECISION NOT NULL — totale netto atteso (dopo sconto globale)
- `verification_status` TEXT NOT NULL DEFAULT 'pending_verification' — stati: pending_verification, verified, mismatch_detected, auto_corrected, correction_failed
- `verified_at` TIMESTAMPTZ — timestamp della verifica
- `verification_notes` TEXT — dettagli discrepanze trovate (per Phase 2+)
- `created_at` TIMESTAMPTZ NOT NULL DEFAULT NOW()
- UNIQUE constraint su (order_id, user_id)

**Tabella 2: `agents.order_verification_snapshot_items`** (righe articoli)
- `id` SERIAL PRIMARY KEY
- `snapshot_id` INTEGER NOT NULL REFERENCES agents.order_verification_snapshots(id) ON DELETE CASCADE
- `article_code` TEXT NOT NULL
- `article_description` TEXT
- `quantity` DOUBLE PRECISION NOT NULL
- `unit_price` DOUBLE PRECISION NOT NULL
- `line_discount_percent` DOUBLE PRECISION — sconto riga (nullable)
- `expected_line_amount` DOUBLE PRECISION NOT NULL — importo netto atteso per riga
- `created_at` TIMESTAMPTZ NOT NULL DEFAULT NOW()

**Indici:**
- `idx_verification_snapshots_order` su (order_id, user_id)
- `idx_verification_snapshots_status` su (verification_status)
- `idx_verification_snapshot_items_snapshot` su (snapshot_id)

Seguire il pattern delle migration esistenti (SQL puro, no ORM). Usare schema `agents.` come tutte le tabelle per-agente.

Creare un nuovo file repository dedicato alla verifica ordini. Questo file conterrà le funzioni di accesso dati per il sistema di verifica (Phase 1-6).

**Funzione `saveOrderVerificationSnapshot`:**
- Parametri: `tx: DbPool` (transazione), `orderId: string`, `userId: string`, `data: SnapshotData`
- `SnapshotData` type con: `globalDiscountPercent?: number`, `expectedGrossAmount: number`, `expectedTotalAmount: number`, `items: SnapshotItem[]`
- `SnapshotItem` type con: `articleCode: string`, `articleDescription: string | null`, `quantity: number`, `unitPrice: number`, `lineDiscountPercent: number | null`, `expectedLineAmount: number`

**Logica:**
1. INSERT nella tabella `order_verification_snapshots` con RETURNING id
2. Batch INSERT in `order_verification_snapshot_items` usando lo stesso pattern di placeholder parametrizzati di submit-order.ts (linee 153-191)
3. Usare `ON CONFLICT (order_id, user_id) DO UPDATE` per gestire re-submit dello stesso ordine (sovrascrivere snapshot precedente, cancellare items vecchi via CASCADE)

**Funzione `getOrderVerificationSnapshot`** (per Phase 2, ma creiamola ora):
- Parametri: `pool: DbPool`, `orderId: string`, `userId: string`
- Ritorna: snapshot header + array di items, oppure null se non esiste
- Query JOIN tra le due tabelle

**Export:** Esportare entrambe le funzioni + i tipi `SnapshotData` e `SnapshotItem`.

Seguire le convenzioni del codebase:
- `import type { DbPool }` per type-only imports (C-6)
- Funzioni pure che accettano DbPool come primo parametro (D-1)
- Parametrized queries per prevenire SQL injection (D-2)

    ```sql
    -- Migration 009: Sync checkpoints for run/skip decisions
    -- Tracks per-type sync status to enable threshold-based scheduling

    CREATE TABLE IF NOT EXISTS shared.sync_checkpoints (
      sync_type TEXT PRIMARY KEY,
      status TEXT NOT NULL DEFAULT 'idle'
        CHECK (status IN ('idle', 'in_progress', 'completed', 'failed')),
      items_processed INTEGER NOT NULL DEFAULT 0,
      started_at BIGINT,
      completed_at BIGINT,
      error TEXT,
      updated_at BIGINT NOT NULL DEFAULT (EXTRACT(EPOCH FROM NOW()) * 1000)::BIGINT
    );
    ```

    - sync_type is PK: one checkpoint per sync type (sync-customers, sync-products, etc.)
    - Use BIGINT for timestamps (epoch ms, consistent with sync-sessions)
    - status tracks current state of each sync operation
    - No page-level columns (current_page, total_pages, last_successful_page) — ALL syncs are PDF-based single-shot operations, no pagination to resume from
    - items_processed tracks volume for reporting

  <action>
    Create repository following existing patterns from sync-sessions.ts:

    1. **Types:**
       ```typescript
       type SyncCheckpoint = {
         syncType: string;
         status: 'idle' | 'in_progress' | 'completed' | 'failed';
         itemsProcessed: number;
         startedAt: number | null;
         completedAt: number | null;
         error: string | null;
         updatedAt: number;
       };

       type ResumePoint = {
         action: 'run' | 'skip';
         reason: string;
       };
       ```

    2. **getResumePoint(pool, syncType) -> ResumePoint:**
       Threshold logic (all syncs are single-shot PDF operations, no page-level resume):
       - If no checkpoint exists: run (first time)
       - If status is 'idle': run (reset or never started)
       - If status is 'in_progress' and startedAt < 30 min ago: skip (sync is currently running — prevent duplicate)
       - If status is 'in_progress' and startedAt >= 30 min ago: run (stale lock — previous sync crashed without cleanup)
       - If status is 'failed': run (retry after failure)
       - If status is 'completed' and completedAt < 1 hour ago: skip (recently completed, too soon)
       - If status is 'completed' and completedAt >= 1 hour ago: run (stale enough to re-sync)

       Note: the 24h "forced restart" from master design is unnecessary here because ALL syncs are full re-syncs (no incremental/delta at sync service level). The 1h threshold prevents re-running too frequently while the scheduler's 10min/30min intervals handle normal cadence.

    3. **startSync(pool, syncType):**
       - UPSERT: INSERT ON CONFLICT (sync_type) DO UPDATE
       - Set status='in_progress', items_processed=0, started_at=Date.now(), clear error and completed_at

    4. **completeSync(pool, syncType, itemsProcessed):**
       - UPDATE status='completed', items_processed, completed_at=Date.now(), updated_at

    5. **failSync(pool, syncType, error):**
       - UPDATE status='failed', error, updated_at

    6. **resetCheckpoint(pool, syncType):**
       - UPDATE status='idle', reset counters, clear timestamps

    7. **getCheckpointStats(pool) -> SyncCheckpoint[]:**
       - SELECT all checkpoints ordered by updated_at DESC

    8. **Create comprehensive spec** following sync-sessions.spec.ts pattern:
       - Test getResumePoint: no checkpoint → run
       - Test getResumePoint: idle → run
       - Test getResumePoint: in_progress < 30min → skip
       - Test getResumePoint: in_progress >= 30min → run (stale lock)
       - Test getResumePoint: failed → run
       - Test getResumePoint: completed < 1h → skip
       - Test getResumePoint: completed >= 1h → run
       - Test startSync UPSERT creates and updates
       - Test completeSync/failSync state transitions
       - Test resetCheckpoint clears all state
       - Use mock DbPool following existing pattern

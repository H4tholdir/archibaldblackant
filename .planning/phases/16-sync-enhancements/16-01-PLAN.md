    ```sql
    -- Migration 006: Sync checkpoints for page-level resume
    -- Enables resumable syncs that skip already-processed pages after interruption

    CREATE TABLE IF NOT EXISTS shared.sync_checkpoints (
      sync_type TEXT PRIMARY KEY,
      status TEXT NOT NULL DEFAULT 'idle'
        CHECK (status IN ('idle', 'in_progress', 'completed', 'failed')),
      current_page INTEGER NOT NULL DEFAULT 0,
      total_pages INTEGER,
      items_processed INTEGER NOT NULL DEFAULT 0,
      last_successful_page INTEGER NOT NULL DEFAULT 0,
      started_at BIGINT,
      completed_at BIGINT,
      error TEXT,
      updated_at BIGINT NOT NULL DEFAULT (EXTRACT(EPOCH FROM NOW()) * 1000)::BIGINT
    );
    ```

    - sync_type is PK because there's only one checkpoint per sync type at a time
    - Use BIGINT for timestamps (epoch ms, consistent with sync-sessions)
    - status tracks the current state of the sync operation
    - last_successful_page enables resume from last known good page
  </action>

  <files>archibald-web-app/backend/src/db/repositories/sync-checkpoints.ts, archibald-web-app/backend/src/db/repositories/sync-checkpoints.spec.ts</files>
  <action>
    Create repository following existing patterns from sync-sessions.ts:

    1. **Types:**
       ```typescript
       type SyncCheckpoint = {
         syncType: string;
         status: 'idle' | 'in_progress' | 'completed' | 'failed';
         currentPage: number;
         totalPages: number | null;
         itemsProcessed: number;
         lastSuccessfulPage: number;
         startedAt: number | null;
         completedAt: number | null;
         error: string | null;
         updatedAt: number;
       };

       type ResumePoint = {
         action: 'resume' | 'skip' | 'restart';
         startPage: number;
         reason: string;
       };
       ```

    2. **getResumePoint(pool, syncType) -> ResumePoint:**
       Threshold logic:
       - If status is 'in_progress' or 'failed': resume from lastSuccessfulPage + 1
         (the sync was interrupted; pick up where we left off)
       - If status is 'completed' and completedAt < 1 hour ago: skip
         (recently completed; no need to re-run)
       - If status is 'completed' and completedAt >= 1 hour ago but < 24 hours: skip
         (still recent enough; scheduled syncs handle this)
       - If status is 'completed' and completedAt >= 24 hours ago: restart from page 1
         (stale data; full re-sync needed)
       - If status is 'idle' or no checkpoint exists: restart from page 1

    3. **startSync(pool, syncType):**
       - UPSERT checkpoint: set status='in_progress', current_page=0, items_processed=0, started_at=Date.now(), clear error and completed_at

    4. **updateProgress(pool, syncType, page, totalPages, itemsProcessed):**
       - UPDATE current_page, total_pages, items_processed, last_successful_page=page, updated_at

    5. **completeSync(pool, syncType):**
       - UPDATE status='completed', completed_at=Date.now(), updated_at

    6. **failSync(pool, syncType, error):**
       - UPDATE status='failed', error, updated_at (keep last_successful_page for resume)

    7. **resetCheckpoint(pool, syncType):**
       - UPDATE status='idle', reset all counters

    8. **getSyncStats(pool) -> SyncCheckpoint[]:**
       - SELECT all checkpoints ordered by updated_at DESC

    9. **Create comprehensive spec:**
       - Test getResumePoint with each threshold scenario
       - Test startSync creates/updates checkpoint
       - Test updateProgress tracks page progression
       - Test completeSync/failSync state transitions
       - Test resetCheckpoint clears state
       - Use mock DbPool following sync-sessions.spec.ts pattern

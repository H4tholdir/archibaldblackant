<task type="auto">
  <name>Task 1: Endpoint parity audit — extract master inventory and compare with branch</name>
  <files>archibald-web-app/backend/src/parity-audit.spec.ts</files>
  <action>
Create a Vitest test file that performs a systematic endpoint parity audit:

1. Extract all endpoint definitions from master's index.ts using `git show master:archibald-web-app/backend/src/index.ts`. Parse every `app.(get|post|put|patch|delete)("/api/...")` call to build a master endpoint inventory: `{ method, path, requiresAuth }`.

2. For the branch, use supertest against createApp(mockDeps) to probe each master endpoint. For each endpoint:
   - Send a request with the same HTTP method
   - Verify it does NOT return 404 (endpoint exists)
   - If master required auth, verify branch returns 401 without token
   - If master was unauthenticated, verify branch returns non-401

3. Structure as a parameterized test using `test.each()` with the master endpoint list:
   ```
   describe('endpoint parity audit', () => {
     test.each(masterEndpoints)('$method $path exists in branch', ...)
   })
   ```

4. For endpoints that are intentionally different (renamed/consolidated), add explicit mapping entries with comments explaining why. Known mappings:
   - master's `/api/customers/search` → branch's `/api/customers` (search via query param)
   - master's `/api/products/search` → branch's `/api/products/search` (same)
   - master's inline sync routes → branch's `/api/sync/*` router
   - master's inline order routes → branch's `/api/orders/*` + `/api/operations/*` routers

5. Report at the end: total master endpoints, matched, intentionally changed, missing.

Do NOT include endpoints from master's imported sub-routers (delta-sync, bot, warehouse-routes, fresis-history-routes, admin-routes, sync-routes, share-routes) — only the inline endpoints in index.ts. The sub-routers already have their own spec files.

IMPORTANT: The audit test file should be self-contained — it parses master's source using git, not hardcoded strings. If git parsing is too complex, hardcode the master endpoint list as a const array with a comment referencing the source.
  </action>

  <done>Parity audit test passes. Every master inline endpoint either exists at the same path in branch, has an explicit mapping entry, or is documented as intentionally removed/consolidated. Zero unexplained 404s.</done>

  <action>
Extend server.spec.ts with a new `describe('cross-flow integration')` block that tests multi-step sequences through the full Express app:

1. **Auth flow test**: POST /api/auth/login with mock credentials → verify response has token → use token on protected endpoint (GET /api/customers) → verify 200 (not 401). This requires wiring the mock verifyToken in deps to return a valid user payload when called with the token from generateJWT.

2. **Operation enqueue + status flow**: POST /api/operations/enqueue with valid auth → verify returns jobId → GET /api/operations/:jobId/status with same auth → verify returns status object. Mock queue.enqueue to return 'job-123', mock queue.getJobStatus to return a completed job shape.

3. **Unauthenticated access control**: Verify that all protected route groups return 401 without token. Test each mount point: /api/customers, /api/products, /api/orders, /api/operations/stats, /api/admin/users, /api/warehouse/boxes, /api/fresis-history, /api/widget/dashboard-data, /api/users/me/target, /api/subclients. Use test.each for brevity.

4. **Public endpoints access**: Verify that public endpoints return non-401 without token: /api/health, /metrics, /api/timeouts/stats, /api/sync/quick-check, /api/share/pdf/nonexistent.

Use the existing createMockDeps() function. For the auth flow, configure deps.verifyToken to return `{ userId: 'user-1', username: 'agent1', role: 'agent' }` when the token from generateJWT is passed in, and deps.generateJWT to return a real token via the imported generateJWT function.

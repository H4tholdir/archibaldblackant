<task type="auto">
  <name>Task 1: Response shape regression tests for critical API endpoints</name>
  <files>archibald-web-app/backend/src/response-shapes.spec.ts</files>
  <action>
Create a dedicated test file that validates the response shape (structure, not values) of the most critical API endpoints. These are the endpoints the frontend actively calls:

1. **Auth endpoints:**
   - POST /api/auth/login → success: `{ success: true, token: string, user: { id, username, fullName, role } }`
   - POST /api/auth/login → failure: `{ success: false, error: string }`
   - GET /api/auth/me → `{ success: true, user: { id, username, fullName, role } }`

2. **Operations endpoints:**
   - POST /api/operations/enqueue → `{ success: true, jobId: string }`
   - GET /api/operations/:jobId/status → `{ success: true, job: { id, status, type, progress?, result?, error? } }` or `{ success: false, error: string }` when not found
   - GET /api/operations/stats → `{ success: true, stats: { waiting, active, completed, failed, delayed, prioritized } }`

3. **Customer endpoints:**
   - GET /api/customers → `{ success: true, data: Array<{ profile, name, ... }> }` or similar
   - GET /api/customers/sync-status → `{ success: true, data: { count, lastSync } }`

4. **Product endpoints:**
   - GET /api/products → `{ success: true, data: Array }`
   - GET /api/products/sync-status → `{ success: true, data: { count, lastSync } }`

5. **Sync endpoints:**
   - GET /api/sync/quick-check → shape validation (no auth required)

For each endpoint, use supertest against createApp(mockDeps) with appropriate auth tokens. Assert the response shape using `expect(response.body).toEqual(expect.objectContaining({ ... }))` with `expect.any(String)`, `expect.any(Number)`, `expect.any(Array)` for value-independent shape checking.

Configure mock deps to return realistic-shaped data (not empty). For example, mock pool.query to return rows with customer/product shapes when testing list endpoints.

Document any intentional shape differences from master with inline comments referencing the phase decision that changed it (e.g., "// Phase 06-01: data.job instead of data.data").
  </action>
  <verify>npm test --prefix archibald-web-app/backend -- --run src/response-shapes.spec.ts passes</verify>
  <done>Response shape regression tests pass for all critical endpoints. Shape assertions use structure-only matching (expect.any). Intentional differences documented with phase references.</done>
</task>

<task type="auto">
  <name>Task 2: Frontend API contract verification</name>
  <files>archibald-web-app/frontend/src/api/api-contracts.spec.ts</files>
  <action>
Create a frontend test that verifies the API call functions match the expected response shapes. This tests the CLIENT side of the contract:

1. Read the frontend API files: src/api/auth.ts, src/api/operations.ts, src/api/customers.ts, src/api/products.ts.

2. For each API function, verify:
   - The URL path used matches the branch's backend routes (no legacy paths)
   - The expected response destructuring matches what the backend returns
   - No references to deprecated paths (/api/orders/status, /api/queue/stats, etc.)

3. Write tests that mock fetch (using vi.fn()) and verify each API function:
   - Calls the correct URL
   - Uses the correct HTTP method
   - Sends the correct headers (Authorization: Bearer)
   - Handles the response shape correctly (destructures the right fields)

Focus on the 5-6 most critical API functions:
   - login (auth.ts)
   - enqueueOperation (operations.ts)
   - getJobStatus (operations.ts)
   - getOperationsDashboard (operations.ts)
   - Customer list/search
   - Product list/search

Use vi.stubGlobal('fetch', ...) to mock fetch and verify call arguments.
  </action>

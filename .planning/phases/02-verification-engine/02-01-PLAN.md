<feature>
  <name>verifyOrderArticles — confronto snapshot vs articoli sincronizzati</name>
  <files>
    archibald-web-app/backend/src/verification/verify-order-articles.ts,
    archibald-web-app/backend/src/verification/verify-order-articles.spec.ts,
    archibald-web-app/backend/src/db/repositories/order-verification.ts
  </files>
  <behavior>
    Funzione pura `verifyOrderArticles(snapshotItems, syncedArticles, globalDiscountPercent?, tolerance?)` → `VerificationResult`

    **Tipi:**
    - `VerificationResult`: { status: 'verified' | 'mismatch_detected', mismatches: ArticleMismatch[] }
    - `ArticleMismatch`: { type, snapshotArticleCode, syncedArticleCode?, field, expected, found }
    - `type`: 'missing' | 'extra' | 'quantity_diff' | 'price_diff' | 'discount_diff' | 'amount_diff'

    **Input snapshotItems** (dal Phase 1 OrderVerificationSnapshot.items):
    - articleCode, quantity, unitPrice, lineDiscountPercent, expectedLineAmount

    **Input syncedArticles** (ParsedArticle dal PDF parser):
    - articleCode, quantity, unitPrice, discountPercent, lineAmount

    **Casi di test (RED phase):**

    1. Ordine corretto → status 'verified', mismatches vuoto
    2. Articolo mancante (nello snapshot ma non nel sync) → mismatch 'missing'
    3. Articolo extra (nel sync ma non nello snapshot) → mismatch 'extra'
    4. Quantità diversa → mismatch 'quantity_diff'
    5. Prezzo diverso → mismatch 'price_diff'
    6. Sconto riga diverso → mismatch 'discount_diff'
    7. Importo riga diverso → mismatch 'amount_diff'
    8. Tolleranza floating point: differenza ≤0.02 su importo → OK (verified)
    9. Tolleranza superata: differenza >0.02 su importo → mismatch
    10. Articoli duplicati (stesso codice, qty diverse): confronto per ordine posizionale
    11. Case-insensitive: "BCS1.000.000" vs "bcs1.000.000" → match
    12. Ordine vuoto (0 articoli snapshot, 0 sync) → verified
    13. Mismatch multipli: più errori nello stesso ordine → tutti rilevati

    **Funzione repository `updateVerificationStatus`:**
    - Parametri: pool, orderId, userId, status ('verified'|'mismatch_detected'|...), notes (JSON stringified mismatches)
    - UPDATE agents.order_verification_snapshots SET verification_status, verified_at, verification_notes
  </behavior>
  <implementation>
    1. Creare directory `src/verification/` per la logica pura di verifica
    2. Implementare `verifyOrderArticles` come funzione pura (no DB, no side effects)
    3. Ordinare entrambi gli array per articleCode (case-insensitive) prima del confronto
    4. Per articoli duplicati (stesso codice): mantenere ordine originale, confrontare per posizione
    5. Usare `Math.abs(a - b) < tolerance` per confronti numerici con tolleranza
    6. Aggiungere `updateVerificationStatus` a `order-verification.ts`
    7. Esportare tutti i tipi per uso in Phase 2 Plan 02 e fasi successive
  </implementation>
</feature>

**Fix approach:**
1. Read AUDIT-FINDINGS.md to get the list of divergences requiring fixes
2. For each critical divergence: fix the branch implementation to match master's behavior
3. For each significant divergence: fix unless it's an intentional architectural improvement
4. Cosmetic divergences: skip (different formatting, naming, etc. that don't affect behavior)
5. Intentional changes: document rationale but do not revert

**Key constraint:** Fixes must preserve the new architecture (DI, PostgreSQL, modular routes). The goal is behavioral parity, not code parity. The branch implementation can look different as long as it produces the same results.

**Codebase constraints:**
- Frontend type-check: `npm run type-check --prefix archibald-web-app/frontend`
- Backend type-check: `npm run build --prefix archibald-web-app/backend`
- Frontend tests: `npm test --prefix archibald-web-app/frontend`
- Backend tests: `npm test --prefix archibald-web-app/backend`

1. Read the master implementation for reference (correct behavior)
2. Read the branch implementation (current behavior)
3. Modify the branch code to match master's behavior while preserving the new architecture

Common fix patterns to expect:
- **Missing validation**: Master validates field X but branch doesn't → add Zod validation
- **Different response shape**: Master returns {field: value} but branch returns {otherField: value} → align response
- **Missing error handling**: Master returns 404 for missing resource but branch returns 500 → add proper error check
- **Different query behavior**: SQLite query returns differently than PostgreSQL equivalent → fix query
- **Missing side effects**: Master emits WebSocket event but branch doesn't → add event broadcast

For each fix:
- Make the minimal change to restore parity
- Do NOT refactor surrounding code
- Do NOT add features not present in master
- Preserve TypeScript strict mode compliance

If a divergence turns out to be an INTENTIONAL improvement in the branch (better error messages, stricter validation, etc.), document it in the summary rather than reverting it.
